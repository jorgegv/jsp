## Task 3: covered-cell compositor optimization

Goal: bring the C covered-cell compositor (`jsp_redraw_covered_cell` +
`jsp_composite_frame_cell` in `lib/jsp_composite.c`) close to SP1's
compositing speed **without** adopting SP1's per-cell memory tables.

### Analysis

JSP's redraw is already structurally identical to SP1's: both are
cell-outer, build each cell in an 8-byte scratch buffer, do one store to
screen, and are differential (JSP's DTT ≈ SP1's invalidation).  Both call
the *same* `sp1_draw_*` rotation kernels.  The gap is purely what SP1
caches in RAM and JSP recomputes:

|                                   | SP1                                           | JSP today                                  |
|-----------------------------------|-----------------------------------------------|--------------------------------------------|
| which sprites cover a cell        | per-cell `slist` (in the 7680 B update array) | scans all N frame-sprites                  |
| sprite graphic pointer for a cell | stored in per-cell `sp1_cs`                   | recomputed (`base + pdc*rowstride + i*cs`) |
| compositor code                   | assembly                                      | C                                          |

SP1 buys speed with ~8 KB of per-*cell* tables.  JSP can recover the same
facts keyed per-*sprite* (bounded by `JSP_SPRITE_REGISTRY_SIZE`) instead
of per-cell (768), i.e. near-zero added memory.  Benchmark reference:
covered path measured at sprite frame B ≈ 5910 frames/1000 vs SP1 ≈ 1600
(`make bench` / `make bench-sp1`).

### Items

- [x] **3.1 — Kill the per-cell address multiplies.**
  `jsp_composite_frame_cell` derived `graph = base + pdc*rowstride +
  i*cs` with two 16-bit `__mul` calls per covering sprite per cell.
  Implemented as **repeated addition** instead of the planned
  precomputed `colbase[]` array: `pdc` and `i` are small, so adding
  beats a multiply for any valid sprite size — and needs **zero** extra
  memory (better than the ~256 B the colbase array would have cost).
- [x] **3.2 — Row-sweep active-sprite set.**  `jsp_redraw_covered_cell`
  rebuilds `jsp_row_active[]` (frame-sprite indices whose `[r0,r1]`
  spans the current row) only when the row changes; per covered cell it
  column-tests just that short list instead of scanning all N sprites.
  Zero memory (a ≤16-byte index array).
  Measured (3.1+3.2 together, `make bench`): sprite frame 5910→5238
  frames/1000 (**−11.4%**); full-screen redraw unchanged. JSP-vs-SP1
  sprite-frame ratio 3.7×→3.3×.
- [x] **3.3 — Rewrite the covered-cell compositor in assembly.**  Folded
  `jsp_redraw_covered_cell` + `jsp_composite_frame_cell` into one asm
  routine (`lib/jsp_covered.asm`): coords/scratch/struct-fields handled
  directly, inline 8-byte seed copy, no sdcc prologue / IX frame. Calls
  the existing `sp1_draw_*` kernels and `jsp_draw_screen_tile_regs`.
  Zero added memory. `jsp_redraw_begin` stays in C.
  Measured (`make bench`): sprite frame B 5238→3900 frames/1000
  (**−25.5%**); full-screen redraw A unchanged (1709). JSP-vs-SP1
  sprite-frame ratio 3.3×→~2.5× (SP1 B ≈ 1582).

## Task 4: convert the remaining hot-path C to assembly

Goal: close more of the JSP-vs-SP1 gap by converting the two pure-C
functions still in the per-frame hot path to assembly, and review/tune
the existing assembly while at it.

### Analysis

Phase B (3877 frames/1000) was instrumented (temporary phases C/D in
`tests/test_redraw_bench.c`) to split the cost:

| Part                                                                                  | Frames | % of B | Status     |
|---------------------------------------------------------------------------------------|--------|--------|------------|
| Deferred move path (`jsp_move_sprite` → `mark_footprint_dirty` → `jsp_dtt_mark_rect`) | 670    | 17%    | C          |
| `jsp_redraw_begin()` per-frame precompute                                             | 303    | 8%     | C          |
| Redraw DTT-walk + covered-cell compositing                                            | 2904   | 75%    | mostly asm |

- The move path costs ~9,400 T-states per sprite move just to update a
  descriptor and OR ~18 dirty bits — SDCC's nested-loop codegen in
  `jsp_dtt_mark_rect`.
- `jsp_redraw_begin` costs ~4,200 T-states per sprite and still emits
  **two real `__mulint` library calls** (`rowstride=(rows+1)*cs`,
  `base=pixels-yrot*(cs>>3)`).  Both multiplies are spurious: `cs` is a
  power of two (8 or 16, from `ismask2`), so asm replaces them with a
  shift by 3/4 and a shift by 0/1.
- Why JSP's non-kernel code is ~5× SP1's: SP1 spends ~8 KB of per-cell
  tables to make per-cell work a *table lookup*; JSP *recomputes*.
  Lookup-vs-recompute is an algorithmic gap, not a coding gap, and is
  the deliberate JSP memory tradeoff — mostly not recoverable.  The
  C-vs-hand-asm part *is* recoverable: that is what Task 4 targets.
  Realistic JSP floor is ~1.5–2× SP1, not parity.

### Items

- [x] **4.1 — Deferred move/mark path → assembly.**  Converted
  `jsp_dtt_mark_rect`, `mark_footprint_dirty`, `jsp_move_sprite`,
  `jsp_draw_sprite`, `jsp_sprite_park` to assembly (`lib/jsp_sprite_defer.asm`);
  the mask2/load1/`_frame` wrappers stay in C.  Also fixed a latent bug:
  `jsp_redraw` left IX (the C frame pointer) trashed via
  `jsp_redraw_covered_cell` — now saved/restored.  Measured (`make bench`):
  sprite frame B 3877→3564 frames/1000 (**−8.1%**); full-screen redraw
  unchanged.
- [ ] **4.2 — `jsp_redraw_begin` → assembly.**  Rewrite the per-frame
  per-sprite precompute in asm; collapse the two `__mulint` calls to
  shifts (`cs` ∈ {8,16}).  ~8% of phase B.
- [ ] **4.3 — Independent review + optimization pass of the assembly.**
  Review `jsp_redraw.asm`, `jsp_covered.asm`, `jsp_screen.asm` and the
  new Task 4 files for tightening opportunities.

Measure every change with `make bench`; reject regressions.  Expected:
landing 4.1+4.2 puts B around ~3200 → ~2.0× SP1.
