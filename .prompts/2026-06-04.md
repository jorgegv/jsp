### Progress notes

- ROOT CAUSE: `jsp_frame.asm` computed `r1 = r0 + rows` unconditionally — the
  vertical analog of `c1 = c0 + (xrot ? cols : cols-1)` but MISSING the
  alignment test. At `yrot==0` (ypos cell-aligned ⟺ bottom scanline mod 8 == 7)
  the sprite spans exactly `rows` cell-rows, so compositing `r0..r0+rows`
  rendered ONE spurious row below it; its line 7 reads past the column's 7-line
  trailing pad → stale pixels (on ZX also a mis-coloured attribute cell). Same
  latent bug on BOTH platforms (shared design).
- FIX (committed `0b8d461`, branch `fix/cpc_bottom_line_artifact` off main):
  `r1 = r0 + (yrot ? rows : rows-1)` in lib/cpc + lib/zx `jsp_frame.asm`;
  `jsp_sprite_defer.asm` dirty-marker tightened to agree. Bonus perf (no
  spurious row at yrot==0): M0 -4%, M1/M1mono -2..3%, M2/FAST ~noise.
- Branch workflow per user: merged `feat/cpc_performance_optimization` → main
  (`59e3965`), then branched the fix off main.
- VERIFIED: CPC all 7 configs AE=0 vs tests/refs/cpc/pixel (their frozen frames
  had no yrot==0 sprite, so unaffected). ZX: 4 refs baked the artifact
  (btt_contents, foreground_tiles, pool_and_colour, sprite_move) → regenerated
  from corrected output; test_sprite_draw + non-sprite tests unchanged.
  Demonstrated with a static M2 blank-bg repro (3 balls, AE=6 buggy→fixed).
- Mode-2 confirmed affected (user's original sighting). **Mode 0: affected
  FUNCTIONALLY** (the spurious row WAS composited — that's the -4% perf win) but
  NOT VISIBLY with the M0 ball asset (AE=0 buggy vs fixed): the MASK artifact is
  only the single overflow scanline, whose visibility depends on sprite
  position, background, and the byte linked right after the asset. M2 blank-bg
  exposes it; M2 grid-bg and M0/M1 static MASK scenes don't.
