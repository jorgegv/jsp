# Tasks

## Task 1: evaluate implicit mask

- Use ultrathink
- New idea for Mode 0 and Mode 1: instead of defining sprites with double bytes (i.e. pixel bytes + mask bytes) we can dedicate one of the available colors as "transparent" (most probably cllor number 0 in both mode 0 and mode 1). This would make sprites occupy half the size (i.e. in Mode 1, por example, sprites would be the same memory size as in ZX. In mode 0 would be double the size of ZX). Tiles would unaffected though.
- In line with the _MONO and _FAST modes, let's call these modes _IMASK (for "implicit mask")
- What would be the impact on the engine, regarding occuppied memory? and in performance?
- Could some special/additional optimizations be done in this mode? We don't have to handle separate masks...
- If there would be an expected performance impact, could you draft a plan for the addition?
- What do you think on this?

### Completion notes

- Evaluated; conclusion: **recommended** for Mode 1 + Mode 0 (skip Mode 2 — with
  pen-0 transparent it collapses to existing `_MONO` semantics).
- Memory: masked sprites 16 B/cell → **8 B/cell** (M1 = ZX size, M0 = 2× ZX);
  +256 B LUT per build. Tiles unaffected.
- Performance: **neutral-to-faster** — the per-pixel mask is derived from the
  graph byte via a uniform 256-byte `imask[g]` LUT, costing the same memory
  accesses as MASK2 (no-rotate) and *fewer* in the rotating path (mask no longer
  shifted/carried).
- Extra optimisations: eliminates left/right mask border-kernel variants
  (transparency falls out of graph=0); zeroed padding = transparent for free.
- Cost: pen 0 unusable as opaque sprite colour (M0 16→15, M1 4→3). Additive to
  MASK2, not a replacement.
- Written up as living design doc: doc/CPC-IMASK-DESIGN.md (per user request).
  Not implemented.

### Progress notes

- Branch `cpc_implicit_mask` created off main.
- Plan written: doc/CPC-IMASK-IMPLEMENTATION-PLAN.md (7 phases, each with a test
  gate). Key decisions: distinct `_jsp_draw_imask{,lb,rb}` kernels + IMASK-aware
  dispatch in jsp_covered.asm; in IMASK builds `cs=8` always and `ismask2` only
  selects kernel (decoupled from the old cs=16 coupling in jsp_frame.asm); LUT
  (256 B, in reserved rottbl block) over in-register derivation; no
  `_IMASK_FAST`, no Mode-2 variant. Corrected design-doc claim: lb/rb border
  kernels are NOT eliminated (they handle graph rotation); what IMASK drops is
  the parallel mask shift + 0xFF border-mask seeding.
- Plan APPROVED 2026-06-08. Implementing in phases (each a small commit):
  - [x] P1 mode plumbing + LUT formula + host test (both modes PASS)
  - [x] P2 LUT data + init (LUT @ 0xB800 M1 / 0xB400 M0)
  - [x] P3 asset tool `cpcgfx.pl --imask` + assets (byte-identical to mask2 graph)
  - [x] P4 kernels + dispatch + frame decouple + type tag
        (cap32: AE=0 vs MASK2 over black, both modes, all xrot phases)
  - [x] P5 C API wrappers (wrapper-API build still AE=0 vs MASK2)
  - [x] P6 Makefile matrix + cpc-tests wiring
  - [x] P7 docs (design doc corrected, ASSETS-FORMAT §7, USAGE, README)
- VERIFICATION (all green): `make cpc-tests` PASS — both _IMASK LUT host tests
  PASS, artifact regression AE=0; `make zx-tests` all AE=0 (no regression from
  shared-header changes); ZX `make build` + both CPC _IMASK configs build clean.
  IMASK kernels pixel-identical to MASK2 over black bg at every xrot phase, both
  modes. All on branch `cpc_implicit_mask` (NOT merged to main yet).
- OPTIMIZATION PASS (baseline tagged `imask-baseline`): kept rottbl page in H,
  LUT via DE, graph ptr in BC; border kernels H-constant. Per-byte mid 130->119T,
  border 110->85T. Still AE=0 both modes, host LUT tests PASS, cpc-tests green.
- HONEST PERF (rigorous: 3 interleaved rounds, boot-free t6000-t2000, crash runs
  discarded): frame-level gain is SMALL — opt IMASK vs MASK2 M1 54.4s vs 56.2s
  (-3.2%), M0 60.9s vs 61.4s (-0.9%) per 4000 redraws; opt vs baseline IMASK
  ~-1%. The per-byte kernel saving is diluted because redraw is dominated by the
  background blit + DTT walk, not the kernel. The earlier single-run "23%" was a
  cap32 noise artifact (corrected in docs). LESSON: cap32 wall-clock needs
  interleaved multi-round sampling + boot-free metric; never trust single runs.
